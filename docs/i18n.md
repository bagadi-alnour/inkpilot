# Internationalization

## Overview

Strings are keyed with **flat, namespaced IDs** (for example `toolbar.bold`, `ai.rewriting`). You set a **locale** for formatting and pass **partial overrides** for translations. English defaults ship with the package and cover every key in `TranslationStrings`.

## `I18nConfig`

```typescript
import type { I18nConfig } from "@writeflow/editor";

const i18n: I18nConfig = {
  locale: "de-DE",
  translations: {
    "toolbar.bold": "Fett",
    "ai.rewrite": "Mit KI umschreiben",
  },
};
```

Pass `i18n` to `<Editor />` and optionally set the top-level `locale` prop. Resolution uses `locale ?? i18n.locale ?? "en"`.

```tsx
<Editor i18n={i18n} locale="de-DE" />
```

## Default English strings

The package provides a full `TranslationStrings` object (default English) so every key resolves without overrides. The type enumerates **all** keys—**59** strings in the current release—spanning toolbar, AI, SEO, SERP, image, and general UI.

Namespaces include:

- `toolbar.*` — formatting actions
- `ai.*` — rewrite UI, tones, intents
- `seo.*` — light signals and panel
- `seo.serp.*` — search preview labels
- `image.*` — upload and alt text
- `general.*` — loading, errors, placeholder

## Translation key structure

Keys are **flat strings**, not nested objects:

```typescript
import type { TranslationKey } from "@writeflow/editor";

const key: TranslationKey = "toolbar.heading1";
```

`TranslationKey` is `keyof TranslationStrings`, so TypeScript autocompletes valid keys when you build locale files.

## Adding a new locale

1. Copy the `TranslationStrings` shape from your editor types (or export a const object typed as `TranslationStrings`).
2. Fill every key, or start with `Partial<TranslationStrings>` and merge onto English defaults (the runtime merges overrides with `defaultStrings`).
3. Pass the result via `i18n.translations` and set `locale` to a BCP 47 tag (e.g. `fr`, `ja-JP`).

```typescript
import type { TranslationStrings } from "@writeflow/editor";

const fr: Partial<TranslationStrings> = {
  "toolbar.bold": "Gras",
  "general.placeholder": "Commencez à écrire...",
};

const i18n = { locale: "fr-FR", translations: fr };
```

## RTL support

The editor does not set text direction by itself. Wrap the editor (or your page) with the correct **`dir`** and **`lang`** for RTL languages:

```tsx
<div dir="rtl" lang="ar">
  <Editor i18n={{ locale: "ar", translations: arStrings }} />
</div>
```

Use the same pattern as standard React/Tiptap apps so selection, toolbar, and paragraphs flow right-to-left.

## Locale-aware number formatting

SEO and metadata UIs benefit from consistent number formatting. Use the configured locale with `Intl`:

```typescript
const formatter = new Intl.NumberFormat(locale, { maximumFractionDigits: 0 });
formatter.format(analysis.score);
```

Keep your surrounding analytics UI on the same `locale` you pass to the editor so scores, word counts, and reading time read consistently.

## Notes

- `TranslationStrings` must include every key at compile time when you aim for full coverage; partial merges are fine at runtime because defaults fill gaps.
- For large apps, consider generating `TranslationStrings` JSON from your translation management tool and importing it per locale.
